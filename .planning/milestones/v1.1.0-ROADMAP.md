# Milestone v1.1.0: Trusted Distribution And Runtime Release

**Status:** SHIPPED 2026-02-28
**Delivered Phases:** 5-6
**Deferred Phases:** 7-8
**Total Delivered Plans:** 6

## Overview

`v1.1.0` shipped the trust and runtime hardening layer on top of the binary-first `v1.0.1` release. The delivered work made official release assets verifiable through checked-in manifests, aggregate checksums, and trust-first GitHub release notes, while also improving runtime diagnostics and local entrypoints for PowerShell 7, WSL, and POSIX shells.

The milestone did not complete its full original roadmap. Batch workflows, aggregate reporting, scanned/OCR diagnostics, and validation/export hooks were deferred into `v1.2.0` after the trust/runtime core reached release quality.

## Delivered Phases

### Phase 5: Trusted Distribution and Install Verification

**Goal:** Make official release assets easier to trust, verify, and adopt without manual Python setup.

Plans:

- [x] 05-01: Finalize signing/notarization validation and release trust checks
- [x] 05-02: Add install/self-test guidance and checksum-first verification flows
- [x] 05-03: Tighten release notes and artifact publishing automation

### Phase 6: Cross-Platform Runtime Ergonomics

**Goal:** Remove platform-specific friction in local usage, especially for PowerShell 7 and WSL users, while hardening the IV&V coverage around those runtime paths.

Plans:

- [x] 06-01: Improve PowerShell-friendly and WSL-friendly wrapper and invocation paths
- [x] 06-02: Harden path, permissions, malformed-input, and platform-specific error messaging
- [x] 06-03: Expand platform verification coverage, test robustness/performance checks, and first-run workflow validation

## Milestone Summary

**Key Decisions:**

- Treat release trust as part of product behavior by requiring checksum verification, `--version`, and `--self-test` before real PDF processing.
- Generate release asset names, checksums, and GitHub release metadata from checked-in manifest scripts to prevent drift.
- Keep POSIX and PowerShell local entrypoints thin by routing them through one shared Python runner.
- Distinguish malformed PDFs, low-text/image-only outcomes, and output-write failures as separate runtime classes.

**Issues Resolved:**

- Added versioned asset naming and aggregate checksum publication to the GitHub release workflow.
- Added generated GitHub release notes that combine the trust matrix with curated release notes.
- Added a PowerShell development entrypoint and aligned local runner guidance across platforms.
- Hardened determinism, reliability, and IV&V coverage without materially increasing test runtime.

**Issues Deferred:**

- Batch processing and aggregate reporting remain follow-on work for `v1.2.0`.
- Explicit scanned/OCR diagnostics beyond low-text signaling remain follow-on work for `v1.2.0`.
- Validation and richer export hooks for downstream review remain follow-on work for `v1.2.0`.

**Technical Debt Incurred:**

- Optional Windows signing and Apple notarization still depend on repository secrets being configured.
- Phase 5 shipped without a dedicated `05-VERIFICATION.md` artifact, which left the milestone audit with a partial verification gap.
